# Raina 面试材料

## 1. 简历项目描述精简版

Raina 是一个基于 Vue 3、NestJS、LangChain 和 Ollama 的塔罗主题 AI 应用，支持 JWT 鉴权、流式聊天、会话持久化、文件上传与 RAG 检索、今日运势生成和中英文国际化，具备完整 AI 产品原型能力。

## 2. 面试 3 分钟项目介绍稿

面试官您好，我想介绍一下我做的项目 Raina。

这个项目是一个塔罗主题的 AI Web 应用，我希望它不是单纯接一个大模型接口做聊天框，而是尽量接近一个完整产品原型。所以我把登录注册、聊天、会话历史、文件上传、RAG、今日运势、牌阵抽取、图鉴浏览和国际化都整合到了一起。

技术上，前端我是用 Vue 3、Pinia、Vue Router、Tailwind CSS 来搭建，后端用 NestJS、Knex 和 MySQL。AI 能力这部分，我用 LangChain 去组织模型调用，用 Ollama 作为本地模型服务，聊天采用 SSE 流式输出，保证用户能边生成边看到内容。为了支持基于用户文档问答，我又加了文件上传和 RAG 检索流程，后端会对文档切分、生成 embedding，再把检索结果拼到 prompt 中。

这个项目里我比较重视的不是功能堆叠，而是工程完整性。比如登录后所有核心页面都要求鉴权访问；会话和消息会持久化到数据库，服务重启后还能恢复；前端 API 层统一封装了 token、语言头和错误处理；前后端都用了 Zod 做运行时校验，避免接口字段变化时静默出错；聊天消息列表还用了虚拟列表优化长对话性能。

我觉得这个项目最大的价值有三点。第一，它覆盖了完整前后端链路，不只是一个页面 Demo。第二，它把 AI 能力和产品能力结合起来了，比如流式聊天、RAG、运势生成和图鉴工具。第三，它能体现我对系统设计和边界处理的理解，比如为什么要做鉴权、为什么要做会话持久化、为什么要在前后端都做类型约束。

如果继续迭代，我下一步会优先做向量库持久化、Docker Compose 一键启动，以及更完整的错误监控和部署方案，把它从“高质量原型”继续推进到“更接近生产可运行”的状态。

## 3. 模拟面试场景

使用方式：

- 你先看“面试官提问”
- 再尝试自己回答
- 如果卡住，再参考“你的回答示例”
- 最后看“面试官深挖”准备下一轮追问

### 第 1 轮

面试官提问：

- 你先整体介绍一下 Raina 这个项目。

你的回答示例：

- Raina 是一个塔罗主题的 AI Web 应用，核心目标不是做单一聊天页，而是做完整产品原型。它包括登录注册、AI 流式聊天、会话持久化、文件上传与 RAG、今日运势、牌阵抽取、图鉴浏览和中英文国际化。前端我使用 Vue 3、Pinia、Vue Router、Tailwind CSS，后端使用 NestJS、Knex、MySQL，AI 部分使用 LangChain 和 Ollama。

面试官深挖：

- 你为什么不做一个简单聊天框，而要做这么多产品能力？

你的回答示例：

- 因为单一聊天框更像接口演示，难以体现工程能力。我要证明的不只是会调用模型，而是能把用户体系、数据持久化、流式交互、文档检索和产品体验整合成完整闭环。

### 第 2 轮

面试官提问：

- 这个项目和普通 AI Demo 的区别是什么？

你的回答示例：

- 最大区别是它不是只有模型调用，而是完整覆盖了前后端链路。普通 Demo 往往只有一个输入框和一个输出框，但这个项目有登录鉴权、会话历史、文件上传、RAG、流式输出、国际化和统一错误处理，所以更接近真实产品。

面试官深挖：

- 如果让我只看一个点，就能判断这个项目不是 Demo，你会让我看哪里？

你的回答示例：

- 我会让您看后端的会话持久化和前端的流式聊天链路。因为这两块同时体现了产品连续性和工程复杂度，不是简单拼装 UI 能解决的。

### 第 3 轮

面试官提问：

- 你为什么选 Vue 3 和 NestJS 这套技术方案？

你的回答示例：

- Vue 3 适合快速搭建结构清晰的产品原型，组合式 API 和 Pinia 在状态拆分上比较自然。NestJS 适合把用户、会话、文件、AI 这些模块边界清晰地拆开，Controller、Service、Guard、Module 结构也更适合中型项目维护。

面试官深挖：

- 如果换成 Express 或 Koa 不行吗？

你的回答示例：

- 当然可以做，但随着模块增多，目录边界、依赖关系和类型约束会更难收敛。NestJS 在这个阶段能让我更快把代码组织成清晰结构，而不是把大量逻辑堆在一起。

### 第 4 轮

面试官提问：

- 你为什么要做登录鉴权？

你的回答示例：

- 因为项目里不仅有聊天，还有会话历史、用户上传文档和 RAG，这些都是用户私有数据。如果不做鉴权，就没法做会话隔离和文件隔离，也很难把项目当成真实产品原型来看。

面试官深挖：

- 你这里的鉴权具体是怎么做的？

你的回答示例：

- 后端用 JWT，登录和注册成功后返回 token，前端保存到 localStorage，后续请求统一通过 Authorization 头传递。前端路由守卫会拦截未登录访问，后端通过 AuthGuard 做二次校验。

### 第 5 轮

面试官提问：

- 你为什么做流式聊天？

你的回答示例：

- 因为 AI 产品很看重首字返回时间。如果等模型一次性生成完成再返回，用户会明显感觉系统在卡住。流式输出能让用户更早看到反馈，体验更接近真实 AI 产品。

面试官深挖：

- 为什么选 SSE，不选 WebSocket？

你的回答示例：

- 这个场景本质上是服务端持续向前端推文本片段，单向传输就够了。SSE 在实现和调试上更轻量，非常适合模型输出流；如果以后做更复杂的双向实时交互，再考虑 WebSocket 也不迟。

### 第 6 轮

面试官提问：

- 会话为什么要持久化到数据库？

你的回答示例：

- 如果只放在内存里，服务一重启，用户历史会话就丢了。把会话和消息持久化到 MySQL 后，用户重新进入系统仍能看到历史记录，服务启动时也能把历史消息重新恢复到运行时上下文里。

面试官深挖：

- 那为什么不每次都直接从数据库取上下文，不做内存态？

你的回答示例：

- 我把它拆成两层是为了平衡连续性和运行效率。数据库负责可靠存储，内存 Agent 负责当前运行时上下文，这样既能恢复历史，又不会把所有上下文恢复成本压到每一次请求上。

### 第 7 轮

面试官提问：

- 你这个项目里的 RAG 是怎么做的？

你的回答示例：

- 用户上传文档后，后端会读取文件内容，使用文本切分器做 chunk，再通过 OllamaEmbeddings 生成 embedding，放到 MemoryVectorStore 中。用户提问时，系统先检索最相关的文档片段，再把检索结果拼到 prompt 里，最后交给模型生成答案。

面试官深挖：

- 为什么现在用的是 MemoryVectorStore？

你的回答示例：

- 因为这个阶段优先目标是把完整链路跑通，MemoryVectorStore 集成成本低、调试快，适合原型验证。下一步如果追求更强的可扩展性，我会换成持久化向量库。

### 第 8 轮

面试官提问：

- 既然用了 TypeScript，为什么还要用 Zod？

你的回答示例：

- TypeScript 只能保证编译期类型，不能保证运行时拿到的数据一定合法。接口返回、请求体、数据库字段在运行时都可能出问题，所以我让后端用 Zod 校验参数，前端用 Zod 校验响应，把错误尽量拦截在边界层。

面试官深挖：

- 你能举一个真实遇到的问题吗？

你的回答示例：

- 有一次会话历史接口里 `is_user` 从数据库返回的是 `0/1`，前端 schema 期望的是 boolean，结果直接触发了 ZodError。这个问题就很典型，说明运行时校验能及时暴露前后端契约不一致。

### 第 9 轮

面试官提问：

- 聊天页面为什么还用了虚拟列表？

你的回答示例：

- 因为聊天是天然会累积很多消息的场景。短会话时看不出问题，但长对话一定会有性能压力。如果所有消息都真实渲染，滚动和更新都会越来越慢，所以我用 `@tanstack/vue-virtual` 做了消息虚拟列表。

面试官深挖：

- 这个优化是不是过早了？

你的回答示例：

- 我觉得不是。聊天应用和普通列表不同，它的消息是持续增长的。提前做虚拟列表其实是在针对确定会出现的性能瓶颈做设计，而不是为了炫技。

### 第 10 轮

面试官提问：

- 为什么做国际化？

你的回答示例：

- 因为我想把这个项目做成完整产品原型，而不是只服务单一语言环境的练手项目。国际化不仅影响页面文案，也影响接口 message、错误提示和语言状态管理，所以应该尽早纳入架构。

面试官深挖：

- 你是怎么保证前后端语言一致的？

你的回答示例：

- 前端通过 `vue-i18n` 管理 UI 文案，并在请求头里统一带 `Accept-Language`。后端根据这个请求头返回本地化 message，这样用户切换语言后，页面和接口提示能保持一致。

### 第 11 轮

面试官提问：

- 如果让你继续优化这个项目，你下一步做什么？

你的回答示例：

- 第一优先级我会做向量库持久化、Docker Compose 一键启动和更完整的错误监控，因为这些能力能显著提升项目的可部署性和稳定性。第二优先级会做消息重新生成、文件处理进度、会话搜索和更精细的权限控制。

面试官深挖：

- 为什么你优先做基础设施，而不是先加更多功能？

你的回答示例：

- 因为项目已经具备完整功能闭环，下一阶段最值钱的是把它变成更稳定、更可部署、更可维护的系统。基础设施补齐后，再加功能的收益会更高。

## 7. 压力面试版追问清单

这一部分专门针对更刁钻的深挖问题，适合在你已经能顺畅回答基础问题之后练习。

## 7.1 RAG 相关

- 你为什么现在用 `MemoryVectorStore`，而不是一开始就用持久化向量库？
- 如果服务重启，当前向量检索能力会丢失吗？你怎么解释这个设计缺陷？
- 你怎么证明检索到的片段真的对回答有帮助，而不是无效噪声？
- 如果上传的文档很多，内存向量检索会不会撑爆进程？你准备怎么演进？
- 文档切分的 `chunkSize` 和 `chunkOverlap` 是怎么定的？如果设置不合理会出什么问题？
- 如果检索出来的内容互相冲突，你打算怎么处理？
- 为什么你的 RAG 现在只支持本地文件，不支持对象存储或者远程知识库？
- 如果用户上传的是很脏的文档，比如 OCR 错字很多、格式很乱，RAG 效果会怎样？
- 你怎么区分“模型本身回答不好”和“检索召回不好”？
- 如果面试官要求你把现在的 RAG 升级到可生产部署，你的第一步会做什么？

你练习时建议这样答：

- 先承认当前方案是“原型优先”
- 再说明它解决了什么核心验证问题
- 最后给出演进路径，比如持久化向量库、异步构建索引、召回评估、重排序

## 7.2 SSE 相关

- 为什么你觉得 SSE 比 WebSocket 更适合这个项目，而不是因为你不会用 WebSocket？
- SSE 如果网络断了怎么办？你现在的恢复策略是什么？
- 流式输出过程中如果后端抛错，前端怎么知道应该停止拼接消息？
- 你为什么不直接让前端等完整回答，而要引入流式复杂度？
- 如果同时有很多用户都在流式聊天，SSE 连接数会不会成为瓶颈？
- 你现在的 SSE 实现如何区分普通 chunk、session id 和 error 事件？
- 如果面试官说“HTTP 本身就不是为长连接设计的”，你怎么回应？
- 你怎么保证流式消息最终落库时和前端展示一致？
- 如果用户在流式过程中刷新页面，当前会话状态会怎样？
- 如果要支持消息中断、重试和续传，你现在的设计够不够？

你练习时建议这样答：

- 先说明 SSE 适合“服务端单向持续推文本”
- 再说明当前实现的边界
- 最后补充如果要更复杂能力，再演进到 WebSocket 或消息队列

## 7.3 JWT 相关

- 为什么你用 JWT，而不是服务端 Session？
- JWT 放在 localStorage 会不会有安全风险？
- 如果 token 泄露，你现在有没有失效控制能力？
- 你现在的 JWT 是无状态的，那怎么处理用户主动退出登录后的即时失效？
- 如果 JWT key 改了，旧 token 全失效，这对线上用户有什么影响？
- 你现在的鉴权流程里，前端路由守卫和后端 AuthGuard 分别负责什么？
- 如果面试官追问“为什么不用 HttpOnly Cookie”，你怎么答？
- 你的 JWT 里放了哪些字段，为什么不多放一些用户信息？
- 如果未来要做刷新 token 机制，你会怎么改？
- 你怎么证明鉴权不是只停留在前端跳转，而是后端也真正做了保护？

你练习时建议这样答：

- 先说当前方案适合前后端分离原型
- 再承认 localStorage 方案的边界
- 最后说明如果往生产演进，会引入 refresh token、黑名单、HttpOnly Cookie 或更严的安全策略

## 7.4 Zod 相关

- 既然有 TypeScript，为什么还要多写一层 Zod，会不会重复劳动？
- 如果后端已经校验了，前端为什么还要校验响应？
- Zod 会不会带来性能损耗？
- 你怎么说服团队接受“多一层运行时校验”的成本？
- 如果接口字段特别多，Zod schema 会不会很难维护？
- 你现在是手写 schema，如果后面接口越来越多，如何避免重复定义？
- 如果数据库里有脏数据，Zod 报错之后你会怎么定位根因？
- 你这套做法和 OpenAPI / Swagger 自动生成类型相比，有什么优缺点？
- 如果面试官说“这是不是过度工程化”，你怎么回答？
- 除了接口响应，你觉得项目里还有哪些地方值得加 Zod？

你练习时建议这样答：

- 先强调 TypeScript 是编译期，Zod 是运行时
- 再举真实踩坑例子，比如 `is_user` 的 `0/1`
- 最后讲边界层校验对稳定性和排障效率的价值

## 7.5 虚拟列表相关

- 你为什么在这个项目里就做了虚拟列表，不是消息量还没大到那一步吗？
- 虚拟列表会不会让代码复杂度上升，收益不一定明显？
- 如果消息高度不固定，虚拟列表怎么处理测量问题？
- 你怎么证明这个优化是真的必要，而不是“为了简历好看”？
- 虚拟列表对 SEO、可访问性或者调试有没有副作用？
- 如果以后要支持消息搜索定位，虚拟列表会不会变复杂？
- 流式消息不断增长时，虚拟列表和自动滚动怎么协同？
- 为什么不用最简单的列表渲染，等真有性能问题再改？
- 如果用户消息里有图片、代码块、长 Markdown，虚拟列表还可靠吗？
- 如果面试官让你定量说明收益，你怎么回答？

你练习时建议这样答：

- 先说明聊天场景的消息是确定会增长的
- 再说明虚拟列表是针对长会话瓶颈的预判式优化
- 最后承认当前项目规模可能还不到极限，但这个设计体现了性能意识和可扩展性思维

这一版适合打印。你可以先遮住“参考回答”和“深挖方向”两列，只看问题自己答。

| 轮次 | 面试官可能怎么问                           | 参考回答要点                                                                                                             | 深挖方向                                             |
| ---- | ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------- |
| 1    | 你先整体介绍一下 Raina 这个项目。          | 塔罗主题 AI 产品原型；覆盖登录、聊天、会话、RAG、运势、图鉴、国际化；前端 Vue 3，后端 NestJS，AI 用 LangChain + Ollama。 | 为什么不做简单聊天框，而要做完整产品能力。           |
| 2    | 这个项目和普通 AI Demo 的区别是什么？      | 不是单一聊天页，而是完整前后端链路；有鉴权、持久化、文件上传、RAG、SSE、i18n。                                           | 如果只看一个点，哪个最能证明它不是 Demo。            |
| 3    | 你为什么选 Vue 3 和 NestJS？               | Vue 3 适合产品原型快速迭代；NestJS 适合模块化组织用户、会话、文件、AI 能力。                                             | 为什么不直接用 Express/Koa。                         |
| 4    | 你为什么要做登录鉴权？                     | 会话历史、文件上传、RAG 都是用户私有数据；鉴权是产品原型必需能力。                                                       | JWT 怎么做；前端和后端分别负责什么。                 |
| 5    | 你为什么做流式聊天？                       | 关键是首字返回时间；流式输出能改善等待体验。                                                                             | 为什么选 SSE，不选 WebSocket。                       |
| 6    | 会话为什么要持久化到数据库？               | 内存会丢历史；MySQL 负责可靠存储；服务启动可恢复上下文。                                                                 | 为什么不是每次都只从数据库取，不保留内存态。         |
| 7    | 你这个项目里的 RAG 是怎么做的？            | 文档上传、切分、embedding、向量检索、把召回内容拼进 prompt。                                                             | 为什么是 MemoryVectorStore；如何演进到持久化向量库。 |
| 8    | 为什么用了 TypeScript 还要用 Zod？         | TypeScript 管编译期，Zod 管运行时；前后端边界都需要校验。                                                                | 用 `is_user` 的 `0/1` 实例说明运行时校验价值。       |
| 9    | 聊天页面为什么还用了虚拟列表？             | 长对话一定会累积大量消息；虚拟列表是针对确定瓶颈的性能优化。                                                             | 是否过度设计；如何解释收益。                         |
| 10   | 为什么做国际化？                           | 项目目标是完整产品原型；i18n 影响 UI、message、语言状态和用户体验一致性。                                                | 前后端如何通过 `Accept-Language` 保持一致。          |
| 11   | 如果让你继续优化这个项目，你下一步做什么？ | 先做向量库持久化、Docker Compose、错误监控；再做消息重生、会话搜索等能力。                                               | 为什么先做基础设施，而不是先堆功能。                 |

## 9. 表格版练习方法

- 第一步，只看“面试官可能怎么问”这一列，自己口述回答
- 第二步，对照“参考回答要点”检查是否讲到了核心点
- 第三步，再看“深挖方向”，继续往下展开 30 秒到 1 分钟
- 第四步，把每一行都练成“20 秒版、40 秒版、90 秒版”三种时长
